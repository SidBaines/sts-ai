# src/sts_ai

Core harness package. See the top-level [`CLAUDE.md`](../../CLAUDE.md) for project goals and repo-wide rules.

## Module map

- `lightspeed.py` — `LightspeedHybridEnv`: wraps the simulator, advances through battles, lists/executes actions, summarizes state. Supports two combat modes via `combat_control`: `"search"` (default, hybrid — battles auto-resolved by the built-in C++ search agent) and `"llm"` (full control — each in-combat micro-decision is surfaced to the agent).
- `agents.py` — policies (`first`, `random`, `heuristic`, optional `mlx`/Qwen for Apple Silicon, `vllm` for NVIDIA/CUDA) and the JSON action parser. `MlxQwenJsonAgent` and `VllmJsonAgent` share one public surface (`reseed`/`choose_action`/`choose_actions_batch`/`config`/mutable `enable_thinking`) so either is a drop-in for the serial and batched loops; `VllmJsonAgent` also implements the `GenerationBackend` streaming primitives (`stream_submit`/`stream_poll`/`stream_has_unfinished`/`build_decision_from_text`) used by `streaming_rollout.py`. `VllmJsonAgent` resolves a general **`reasoning_mode`** (`none`/`native`/`prompted`): it probes whether the model's chat template honours `enable_thinking` — models with a native toggle (Qwen3, **Gemma 4**) use it; models without one (Gemma 3, Llama) get reasoning *induced* via the prompt (`render_action_prompt(induce_reasoning=True)`, which routes CoT through `<think>` tags). **The captured reasoning format is model-dependent — `parse_json_action`/`_extract_thinking_with_metadata` handle both:** Qwen3-native and prompted use `<think>…</think>`; **Gemma 4 native uses a `thought` channel** (`<|channel>thought\n…<channel|>`). vLLM's default `skip_special_tokens=True` strips the markers (completion *starts* with `thought\n…`), but **`VllmJsonAgent.preserve_special_tokens` (default = native) now sets `skip_special_tokens=False` for native runs** so the markers survive into `raw_response` (needed for native-thinking SFT + laundering); `--preserve-special-tokens {auto,on,off}` on `run_until`. `_extract_gemma_thought` captures the thought either way and (bounding at the channel close, else the JSON answer) and tags `metadata["reasoning_format"]="gemma_thought"`; without this the `<think>`-only parser recorded `thinking_tokens=0` for Gemma 4 even though it reasoned (~800 tok). See the Gemma benchmark in `docs/throughput_benchmarks.md`. vLLM generation is **fail-soft** (returns `None` on any error → invalid decisions, never an exception) so a batched sweep survives a model crash. vLLM prefix caching is enabled by default (`enable_prefix_caching=True`) and recorded in `config`. `vllm` is an optional extra (`pip install -e '.[vllm]'`); the import is lazy so the unit tier runs without it.
- `rollout.py` — drives the serial env+agent loop and records `DecisionRecord`s as JSONL. Before each decision it runs `glossary.augment(...)` over `state_text` (so the agent sees it *and* it's recorded). Shared, reused-by-the-parallel-path helpers live here: `prepare_decision` (advance + assemble the augmented view), `clamp_action_index`, `build_decision_record` (incl. computed `affordances`), and `build_rollout_meta`/`write_rollout_meta` (the per-rollout `<output>.meta.json` sidecar with outcome + model/config/**framing**/git provenance).
- `parallel_rollout.py` — `run_parallel_rollouts`: cross-rollout batched orchestrator. Advances K independent envs in lockstep and batches their per-decision generations via `agent.choose_actions_batch` (the throughput lever). Reuses `rollout.py`'s helpers so traces are shape-identical to serial. NOTE: batched generation left-pads → token outputs can differ bit-for-bit from the single-prompt path (fine pre-freeze; pin K for frozen runs).
- `streaming_rollout.py` — `run_streaming_rollouts`: vLLM continuous-batching orchestrator. Keeps up to `concurrency` rollout slots in flight, drives the backend step loop synchronously, dispatches each finished request back by `request_id`, and immediately submits that slot's next decision. Reuses `parallel_rollout.py`'s slot lifecycle helpers and `rollout.py`'s record builders, so trace shape stays aligned with serial/parallel. **Hinting (opt-in `hint_cfg`):** each combat decision can run a per-slot `NORMAL→HINTED→LAUNDER` sub-stage machine (stage-suffixed request_ids `:h`/`:l` + per-stage seeds via `seeding.derive_stage_seed`, both **gated on `hint_cfg.enabled` so hints-off is byte-identical**); the single `commit` local is the only decision_index-advance + slot-reset point, every transition re-submits exactly one request (1:1 slot↔in-flight), and retries stay NORMAL-only. Mirrors the serial `rollout._apply_serial_hint` via the same pure `hinting` helpers.
- `affordances.py` — pure-Python eval-support: per combat decision, what the agent *could* have done (`incoming_damage_total`, `max_block_available`, `full_block_possible`, `single_target_lethal_available`, …). Reimplements `calculateCardBlock` + a curated Ironclad block-base table; logged additively as `DecisionRecord.affordances` (combat only; `{}` otherwise). Does not change `state_text`.
- `glossary.py` — pure, hand-authored static reference (`STATUS_DB`, `CARD_DB`, `INTENT_DB`, `RELIC_DB`, `POTION_DB`) + `augment(state_text, legal_actions, phase, map_graph=None)`. The simulator has no card/effect/relic/potion/intent *text*, so this folds it in. **Combat:** each enemy intent is annotated with its effect from `INTENT_DB` — non-attacks become `(no damage; applies 1 Weak to you)` (the old bare `(no attack)` is only the fallback for unknown moves), attacks get a `(also: …)` rider; the statuses an intent references are added to the KEY; a derived `Incoming attack damage this turn: N` note and a 0-energy/nothing-playable note are appended after the board; the trailing `-- KEY --` block defines active statuses, cards in hand, and potions held. **Out of combat:** the KEY defines offered cards, relics held, and potions. Keys must match the serializer's display strings (`Cards.h cardNames`, `PlayerStatusEffects.h playerStatusStrings`, `monsterMoveStrings`, `getRelicName`, `safePotionName`); unknown names are skipped. Derived notes are appended *after* the parsed sections (Piles/Potions) so they don't break `rollout_view`/`affordances` hand/enemy parsing. No simulator dependency. **Map screens:** when `augment` is given a `map_graph` (the env passes `LightspeedHybridEnv.map_graph()`, non-`None` only on `MAP_SCREEN`), `_render_map` replaces the old unparseable ASCII grid with a neutral per-choice summary — each choice's immediate room + the room composition *reachable downstream* (on at least one path) toward the boss, plus a room-type legend. v1 is deliberately **aggregate reachability** (no branch structure / mutually-exclusive info); may be revisited — see `docs/map_representation_handoff.md` §9. The `choose map node x=… room=…` action labels are **unchanged**, so `risk_proxies` is unaffected and the model cross-references `x=`.
- `schemas.py` — dataclasses defining the **on-disk JSONL format**. This is the data contract consumed by future training/eval.
- `prompting.py` — prompt assembly and the framing strings.
- `risk_proxies.py` — deterministic risk-proxy classification/aggregation over recorded decision dicts (pure Python; no simulator). Keys off the semantic action description; tolerant of the legacy `bits=` prefix in pre-2026-06-14 traces.
- `hinting.py` — **pure** (no simulator/vLLM/agent imports) hinted-rollout-with-laundering core. `HintConfig` (off by default), fixed **tactical-truth** hint strings, `detect_mistake` (combat-only: lethal-available-but-not-taken / full-block-possible-under-heavy-incoming, reading `combat["player_cur_hp"]`), `build_hinted_prompt_suffix`/`build_launder_state_text`, `launder_guardrail_ok`, `action_only_raw_response`, `finalize_hinted_decision` (laundered / action-only / drop; corrected paths kept `valid=True`/`retries=0` so the SFT builder keeps them; provenance under `metadata["hint"]`), `mistake_kind_for` (the shared hint→kind map). Hints MUST stay tactical-truth (no risk/strategy words) — the prompt-neutrality invariant, so framed training can reuse the data. Wired into the serial loop (`rollout._apply_serial_hint`) and the streaming stage machine; toggled via `run_rollout.py`/`run_until.py --hints`.
- `lightspeed_import.py` — locates and imports the locally built `slaythespire` module.
- `train/` — **offline filtered-BC / expert-iteration training pipeline** (Stage 6). Pure-Python core, heavy ML libs lazy-imported so the package imports without `mlx_lm`/`trl`/`torch`. `reward.py` → per-trajectory binary act-boss-clear label (`label_trajectories`); `sft_format.py` → skew-free `{prompt, completion}` reconstruction + the **single** `chat_template_probe_hash` (used by both producer and consumer); `dataset_builder.py` → reward-join + filter → examples + manifest; `train_mlx.py` / `train_trl.py` → thin LoRA-trainer wrappers (subprocess `mlx_lm lora` / TRL `SFTTrainer`+PEFT). Driven by `scripts/build_sft_dataset.py` and `scripts/train_policy.py`; adapters are evaluated via `run_until.py --adapter-path`.

## Area gotchas

- **`MlxQwenJsonAgent` needs a large `max_tokens` (default 4096) for reasoning.** A small cap truncates the model mid-thought → no JSON → the decision is invalid and the rollout stops with `agent_invalid` after retries. Free to raise for no-thinking (stops at EOS first). The default lives in `agents.py`/`agent_factory.py`; the eval's `invalid_rate`/`completion_tokens` catch a too-small cap.
- **Parallel and streaming seed at different granularities.** `parallel_rollout` derives one batch seed per lockstep round from the active `(world_seed, rollout_index, decision_index)` members, so changing `batch_size` can change exact token samples. `streaming_rollout` derives each request seed from its own `(world_seed, rollout_index, decision_index)`, making samples independent of batch composition and completion order.
- **The agent sees only the current state.** `choose_action(state_text, legal_actions)` is Markovian — it receives no after-state, reward, or record of combats resolved between decisions. Any "full-history" mode has to be built around this, not assumed.
- **`step()` re-derives the chosen action by index** from a freshly built `getAllActionsInState` list (`lightspeed.py`). This is only safe while that list is deterministic for a given state. If actions ever become stateful/nondeterministic, store and execute the chosen action object instead of re-fetching by index.
- **In combat, `legal_actions()` is de-duplicated (display indices ≠ raw indices).** `_action_views()` collapses byte-identical action descriptions (two copies of the same card from different hand slots, etc. — ~36% of combat decisions had a duplicate) into a contiguous display list and a `display_to_raw` map; `step()` maps the chosen display index back to the **first** raw action with that description before executing. This is correct only because same-named enemy targets are disambiguated in the action text (`-> NAME [enemy i]`), so an identical description really is interchangeable. **Out of combat the list stays 1:1** (no dedup), so OOC indexing/reproducibility is untouched; combat trace shape/reproducibility intentionally changed (call out in frozen runs).
- **Legal actions are recomputed several times per decision** (loop `advance_to_decision` → `legal_actions` → `step`→`raw_actions`). Correct today, but watch for cost and the index-alignment assumption above.
- **`schemas.py` is load-bearing.** Once we start keeping data, changing these dataclasses is a breaking change to every stored trace — see the schema-stability guideline in the top-level CLAUDE.md.

## Training pipeline (`train/`) gotchas

- **Train/inference skew is the central risk** — `sft_format.reconstruct_prompt` MUST mirror the agent's prompt construction exactly: it calls `render_action_prompt` then `apply_chat_template(..., add_generation_prompt=True, enable_thinking=...)`, uses the stored `state_text` **as-is** (never re-`augment`), and the target is the **verbatim** `agent.raw_response`. The caller must derive `enable_thinking=(reasoning_mode=="native")` and `induce_reasoning=(reasoning_mode=="prompted")` from the run's `reasoning_mode` — NOT from the raw `thinking` flag (a *prompted* Gemma-3 model has `thinking=True` but template `enable_thinking=False`).
- **One definition of the chat-template hash.** `sft_format.chat_template_probe_hash(tokenizer, *, enable_thinking)` is the single source of truth, called by both `dataset_builder` (producer) and `train_trl` (consumer). They were briefly split and drifted (consumer omitted `enable_thinking` → defaulted True on Qwen3/Gemma → false skew-guard rejection of every no-thinking dataset). Keep it shared; the consumer reads `enable_thinking` from the manifest.
- **The builder skips records it can't faithfully reproduce or shouldn't train on:** `action_executed=False` (terminal invalid), `agent.valid=False`, and **`agent.retries>0`** (the retry prompt appended an "invalid response" suffix that is NOT in `state_text`, so the prompt can't be reconstructed). Each is counted in the manifest's `skipped_record_counts`.
- **Binary positive = monotonic milestone**, not outcome-at-end: `final_act > min_act` (or VICTORY) counts as an act-boss clear regardless of how the run later ended (truncation/crash after the clear doesn't un-clear it). The sparsity guardrail (`min_positives` → top-floor-quantile fallback) is **opt-in** (`build_sft_dataset --allow-fallback`); it never writes a fallback dataset silently.
- **RWR mode broadens the pool; weighting is resampling, not a weighted loss.** `build_sft_dataset --weighting-mode rwr` bypasses the act-boss-clear filter and keeps ALL valid trajectories, replicating each `m = clamp(round(exp((floor−baseline)/β)),0,max_multiplier)` times (`reward.rwr_multiplicities`, deterministic, median baseline). Neither trainer backend supports per-example loss weights, so resampling is the only backend-agnostic route — `β→0` ≈ a hard top-half-by-floor filter (NOT identical to `--weighting-mode filter`), `β→∞` ≈ uniform BC. The sparsity-fallback guardrail is **filter-mode only** (N/A in RWR — losers are intentionally kept).
- **`assistant_turn_content` is a verbatim passthrough today, and the action-only hint fallback's `reasoning_format` tag lies.** The SFT-target reconstruction (`sft_format.assistant_turn_content`) returns `raw_response` unchanged for every `reasoning_format`; `reasoning_format` is plumbed but unused (the round-trip-gated home for *future* marker normalization). One record type to mind when that future work lands: a **laundering action-only fallback** (`finalize_hinted_decision`) emits bare JSON with NO `<|channel|>` markers but COPIES `metadata["reasoning_format"]="gemma_thought"` from the hinted decision — tag and content disagree. Harmless while passthrough; a normalization keyed on the tag must NOT assume marker-bearing content for fallback records. Locked by tests in `test_sft_format.py`/`test_hinting.py`.

## In-combat (full-control) mode

- **`self.bc` is the phase signal.** In `combat_control="llm"`, `advance_to_decision` constructs a `BattleContext` and holds it on `self.bc` while an in-combat decision is pending; `phase()` returns `"combat"` iff `self.bc is not None`. The four decision methods (`raw_actions`/`legal_actions`/`describe_state`/`step`) branch on `self.bc` and use `self.bc` (not `self.gc`) as the action context. `step` captures the context **before** executing, since a battle-ending action clears `self.bc` inside the follow-up `advance_to_decision`.
- **The engine drains itself.** `BattleContext.init` and `BattleAction.execute` both call the C++ `executeActions()` internally, so Python never calls `execute_actions` directly — after either, the state is already at the next player decision or a decided `outcome`.
- **Only `PLAYER_NORMAL` and `CARD_SELECT` are player-decision input states.** Every other `InputState` (the `SELECT_ENEMY_ACTIONS`/`SHUFFLE_*`/`FILL_RANDOM_*` "random" family, etc.) is resolved inside `executeActions` and never surfaces as a decision. The C++ enumerator (`BattleScumSearcher2::enumerateActionsForNode`, reused by the binding's `enumerate_battle_actions`) only handles those two — full parity with the built-in search agent and `BattleSimulator`. Other player-choice states (`CHOOSE_STANCE_ACTION`, `SCRY`, `CHOOSE_*_CARDS`, `GAMBLE`, …) are reachable only via specific cards/potions that are out of milestone-1 scope; if one is hit, `enumerate_battle_actions` **throws** (a clear `simulator_error` naming the state) rather than returning an empty list that would silently stop the rollout `no_legal_actions`. Handling them is the natural next extension.
- **Combat-specific state rides in `state["combat"]`** (turn, input_state, battle_outcome, player hp/block/energy, `undefined_behavior_evoked`, and a structured `enemies` list — index/name/cur_hp/max_hp/block/intent/alive plus the sim-computed `intent_damage`/`intent_hits`/`strength`/`vulnerable`/`weak`/`poison` from `BattleContext.enemies()`); the only schema change was the additive `DecisionRecord.phase`. Combat actions reuse `LegalAction` (`bits`/`description`) and the `{reasoning, action_index}` agent JSON unchanged.
- **Two combat-trace quirks to know when consuming records.** (1) During combat, the **top-level `state["cur_hp"]` is the map (`GameContext`) HP and can be stale**; the authoritative in-combat value is `state["combat"]["player_cur_hp"]`. (2) The structured `enemies` list **includes dead enemies** (`alive: false`), but `state_text`/`describeBattleState` lists **only living ones** — so the hand/enemy text is not 1:1 with the structured enemy array. The hand, pile counts, player powers, and max-energy appear **only in `state_text`** (not in `state["combat"]`); `rollout_view.to_combat_view` parses them from there.
- **`undefined_behavior_evoked` is latched** at env level (`self._undefined_behavior_evoked`) because `exit_battle` clears `self.bc` before the after-state is recorded; it is surfaced as a top-level `summary()` field (OR'd with any in-progress battle) so a UB raised by a battle-ending action is not lost.
- **Combat reuses the neutral framing.** No prompt/agent-protocol change: combat decisions flow through the same `choose_action(state_text, legal_actions)` with combat-aware `state_text` and action descriptions. Card-play and potion actions name the card/potion and target; `SINGLE_CARD_SELECT` actions resolve and name the card from the task's source pile (`describeCardSelectChoice`, mirroring the engine's enumeration). Intent labels in the serializer are the engine's raw move strings (e.g. `ACID_SLIME_S_LICK`).
- **The serializers carry sim-computed numbers (binding-side enrichment).** No card/relic/potion *text* exists in the simulator, but the binding (`bindings/slaythespire.cpp`) computes and appends decision-relevant numbers so the model sees what a player gets by hovering:
  - **Enemy intent damage** — `intent <MOVE> (deal 11)` or multi-hit `(deal 32 = 16 x2)`, gated on `Monster::isAttacking()` (non-attacks show no number). **Pre-block, per-hit×count**, via `getMoveBaseDamage`/`calculateDamageToPlayer`. Non-zero enemy statuses (`Strength/Vulnerable/Weak/Poison`) follow on the same line.
  - **Card attack damage** — `play Strike (cost 1) -> Jaw Worm (deal 9)`, target-correct via `calculateCardDamage`. Only for plain `ATTACK` cards; **X-cost and special-formula cards (Body Slam, Heavy Blade, Perfected Strike, Rampage, Searing Blow, Fiend Fire) are deliberately skipped** because `getBaseDamage + calculateCardDamage` would mis-report them. Multi-hit counts come from the curated `getHitCount(id, upgraded)` helper (the static twin of `getBaseDamage`; Ironclad-scoped — keep both in sync with the card pool).
  - **OOC card choices** — reward/shop/card-select append ` [Type, Rarity]` (e.g. `take card Offering [Skill, Uncommon]`) via `Card::getType()`/`getRarity()`.
- **Same-named enemy targets are disambiguated.** `describeBattleAction` appends `[enemy i]` after a play action's target **only when another living enemy shares that name** (3 Sentries, split slimes — ~8k combat decisions). This both lets the model target a specific one and makes the Python combat de-dup safe. Two parsers consume the suffix: `rollout_view` (`_ENEMY_IDX_RE`, parses the index as the authoritative target) and `affordances` (strips it so name-keyed lethal matching still works); `risk_proxies` is unaffected (out-of-combat only).
- **The act map is exposed as a graph, not ASCII.** `describeGameState` **no longer dumps `gc.map->toString`** (the ASCII grid was unparseable by LLMs — see `docs/map_representation_handoff.md`). Instead the binding adds `GameContext.map_graph()` → `{"cur_y", "nodes":[{"x","y","room","edges"}]}` (`edges` = child x-indices in row `y+1`; existing nodes only; `cur_y` = current row, `-1` pre-first-row / `14` boss). `LightspeedHybridEnv.map_graph()` returns it on a `MAP_SCREEN` (else `None`); `glossary._render_map` turns it into the per-choice text. **If you change `map_graph`'s shape, update `glossary._render_map` + its tests together.**
- **OOC serializer carries player-facing labels.** `describeGameState` emits `Your HP:` (was bare `HP:`, misread as the boss's HP — the `SimpleHeuristicAgent`/`rollout_view` regexes still match the `HP:`/`boss <NAME>` substrings), `end-of-act boss <NAME>`, and on `CARD_SELECT` a `cardSelectPurpose(gc.info.selectScreenType)` line ("Choose a card to REMOVE/UPGRADE/TRANSFORM/…") instead of the bare `Card select: 0/1`.
- **Two parsers are coupled to these strings — change them together.** (1) `rollout_view._PLAY_RE` tolerates the trailing `(deal …)` so the target group doesn't swallow it. (2) `risk_proxies.classify_decision` strips a trailing `[…]` tag before matching the reward card name against `SELF_DAMAGE_CARDS`. If you change the `(deal …)`, `[Type, Rarity]`, or `[enemy i]` format in the binding, update both (and `_ENEMY_IDX_RE`/the affordances strip above). The enrichment **intentionally changes `state_text`/action-description content** (so LLM/heuristic trajectories shift); it does not affect the hybrid search agent. No `schemas.py` change (combat rides in free-form `state["combat"]`).
