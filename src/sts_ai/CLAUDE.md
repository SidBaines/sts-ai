# src/sts_ai

Core harness package. See the top-level [`CLAUDE.md`](../../CLAUDE.md) for project goals and repo-wide rules.

## Module map

- `lightspeed.py` ‚Äî `LightspeedHybridEnv`: wraps the simulator, advances through battles, lists/executes actions, summarizes state. Supports two combat modes via `combat_control`: `"search"` (default, hybrid ‚Äî battles auto-resolved by the built-in C++ search agent) and `"llm"` (full control ‚Äî each in-combat micro-decision is surfaced to the agent).
- `agents.py` ‚Äî policies (`first`, `random`, `heuristic`, optional `mlx`/Qwen) and the JSON action parser.
- `rollout.py` ‚Äî drives the serial env+agent loop and records `DecisionRecord`s as JSONL. Before each decision it runs `glossary.augment(...)` over `state_text` (so the agent sees it *and* it's recorded). Shared, reused-by-the-parallel-path helpers live here: `prepare_decision` (advance + assemble the augmented view), `clamp_action_index`, `build_decision_record` (incl. computed `affordances`), and `build_rollout_meta`/`write_rollout_meta` (the per-rollout `<output>.meta.json` sidecar with outcome + model/config/**framing**/git provenance).
- `parallel_rollout.py` ‚Äî `run_parallel_rollouts`: cross-rollout batched orchestrator. Advances K independent envs in lockstep and batches their per-decision generations via `agent.choose_actions_batch` (the throughput lever). Reuses `rollout.py`'s helpers so traces are shape-identical to serial. NOTE: batched generation left-pads ‚Üí token outputs can differ bit-for-bit from the single-prompt path (fine pre-freeze; pin K for frozen runs).
- `affordances.py` ‚Äî pure-Python eval-support: per combat decision, what the agent *could* have done (`incoming_damage_total`, `max_block_available`, `full_block_possible`, `single_target_lethal_available`, ‚Ä¶). Reimplements `calculateCardBlock` + a curated Ironclad block-base table; logged additively as `DecisionRecord.affordances` (combat only; `{}` otherwise). Does not change `state_text`.
- `glossary.py` ‚Äî pure, hand-authored static reference (`STATUS_DB`, `CARD_DB`) + `augment(state_text, legal_actions, phase)`. The simulator has no card/effect *text*, so this folds it in: inline `(no attack)` labels on non-attacking enemy intents and a trailing `-- KEY --` block defining active statuses (with duration semantics) and the cards in play. Keys must match the serializer's display strings (`Cards.h cardNames`, `PlayerStatusEffects.h playerStatusStrings`); unknown names are skipped. No simulator dependency.
- `schemas.py` ‚Äî dataclasses defining the **on-disk JSONL format**. This is the data contract consumed by future training/eval.
- `prompting.py` ‚Äî prompt assembly and the framing strings.
- `risk_proxies.py` ‚Äî deterministic risk-proxy classification/aggregation over recorded decision dicts (pure Python; no simulator). Keys off the semantic action description; tolerant of the legacy `bits=` prefix in pre-2026-06-14 traces.
- `lightspeed_import.py` ‚Äî locates and imports the locally built `slaythespire` module.

## Area gotchas

- **`MlxQwenJsonAgent` needs a large `max_tokens` (default 4096) for reasoning.** A small cap truncates the model mid-thought ‚Üí no JSON ‚Üí the decision is invalid and the loop falls back to action 0 (degenerate "always first action"). Free to raise for no-thinking (stops at EOS first). The default lives in `agents.py`/`agent_factory.py`; the eval's `invalid_rate`/`completion_tokens` catch a too-small cap.
- **The agent sees only the current state.** `choose_action(state_text, legal_actions)` is Markovian ‚Äî it receives no after-state, reward, or record of combats resolved between decisions. Any "full-history" mode has to be built around this, not assumed.
- **`step()` re-derives the chosen action by index** from a freshly built `getAllActionsInState` list (`lightspeed.py`). This is only safe while that list is deterministic for a given state. If actions ever become stateful/nondeterministic, store and execute the chosen action object instead of re-fetching by index.
- **Legal actions are recomputed several times per decision** (loop `advance_to_decision` ‚Üí `legal_actions` ‚Üí `step`‚Üí`raw_actions`). Correct today, but watch for cost and the index-alignment assumption above.
- **`schemas.py` is load-bearing.** Once we start keeping data, changing these dataclasses is a breaking change to every stored trace ‚Äî see the schema-stability guideline in the top-level CLAUDE.md.

## In-combat (full-control) mode

- **`self.bc` is the phase signal.** In `combat_control="llm"`, `advance_to_decision` constructs a `BattleContext` and holds it on `self.bc` while an in-combat decision is pending; `phase()` returns `"combat"` iff `self.bc is not None`. The four decision methods (`raw_actions`/`legal_actions`/`describe_state`/`step`) branch on `self.bc` and use `self.bc` (not `self.gc`) as the action context. `step` captures the context **before** executing, since a battle-ending action clears `self.bc` inside the follow-up `advance_to_decision`.
- **The engine drains itself.** `BattleContext.init` and `BattleAction.execute` both call the C++ `executeActions()` internally, so Python never calls `execute_actions` directly ‚Äî after either, the state is already at the next player decision or a decided `outcome`.
- **Only `PLAYER_NORMAL` and `CARD_SELECT` are player-decision input states.** Every other `InputState` (the `SELECT_ENEMY_ACTIONS`/`SHUFFLE_*`/`FILL_RANDOM_*` "random" family, etc.) is resolved inside `executeActions` and never surfaces as a decision. The C++ enumerator (`BattleScumSearcher2::enumerateActionsForNode`, reused by the binding's `enumerate_battle_actions`) only handles those two ‚Äî full parity with the built-in search agent and `BattleSimulator`. Other player-choice states (`CHOOSE_STANCE_ACTION`, `SCRY`, `CHOOSE_*_CARDS`, `GAMBLE`, ‚Ä¶) are reachable only via specific cards/potions that are out of milestone-1 scope; if one is hit, `enumerate_battle_actions` **throws** (a clear `simulator_error` naming the state) rather than returning an empty list that would silently stop the rollout `no_legal_actions`. Handling them is the natural next extension.
- **Combat-specific state rides in `state["combat"]`** (turn, input_state, battle_outcome, player hp/block/energy, `undefined_behavior_evoked`, and a structured `enemies` list ‚Äî index/name/cur_hp/max_hp/block/intent/alive plus the sim-computed `intent_damage`/`intent_hits`/`strength`/`vulnerable`/`weak`/`poison` from `BattleContext.enemies()`); the only schema change was the additive `DecisionRecord.phase`. Combat actions reuse `LegalAction` (`bits`/`description`) and the `{reasoning, action_index}` agent JSON unchanged.
- **Two combat-trace quirks to know when consuming records.** (1) During combat, the **top-level `state["cur_hp"]` is the map (`GameContext`) HP and can be stale**; the authoritative in-combat value is `state["combat"]["player_cur_hp"]`. (2) The structured `enemies` list **includes dead enemies** (`alive: false`), but `state_text`/`describeBattleState` lists **only living ones** ‚Äî so the hand/enemy text is not 1:1 with the structured enemy array. The hand, pile counts, player powers, and max-energy appear **only in `state_text`** (not in `state["combat"]`); `rollout_view.to_combat_view` parses them from there.
- **`undefined_behavior_evoked` is latched** at env level (`self._undefined_behavior_evoked`) because `exit_battle` clears `self.bc` before the after-state is recorded; it is surfaced as a top-level `summary()` field (OR'd with any in-progress battle) so a UB raised by a battle-ending action is not lost.
- **Combat reuses the neutral framing.** No prompt/agent-protocol change: combat decisions flow through the same `choose_action(state_text, legal_actions)` with combat-aware `state_text` and action descriptions. Card-play and potion actions name the card/potion and target; `SINGLE_CARD_SELECT` actions resolve and name the card from the task's source pile (`describeCardSelectChoice`, mirroring the engine's enumeration). Intent labels in the serializer are the engine's raw move strings (e.g. `ACID_SLIME_S_LICK`).
- **The serializers carry sim-computed numbers (binding-side enrichment).** No card/relic/potion *text* exists in the simulator, but the binding (`bindings/slaythespire.cpp`) computes and appends decision-relevant numbers so the model sees what a player gets by hovering:
  - **Enemy intent damage** ‚Äî `intent <MOVE> (deal 11)` or multi-hit `(deal 32 = 16 x2)`, gated on `Monster::isAttacking()` (non-attacks show no number). **Pre-block, per-hit√ócount**, via `getMoveBaseDamage`/`calculateDamageToPlayer`. Non-zero enemy statuses (`Strength/Vulnerable/Weak/Poison`) follow on the same line.
  - **Card attack damage** ‚Äî `play Strike (cost 1) -> Jaw Worm (deal 9)`, target-correct via `calculateCardDamage`. Only for plain `ATTACK` cards; **X-cost and special-formula cards (Body Slam, Heavy Blade, Perfected Strike, Rampage, Searing Blow, Fiend Fire) are deliberately skipped** because `getBaseDamage + calculateCardDamage` would mis-report them. Multi-hit counts come from the curated `getHitCount(id, upgraded)` helper (the static twin of `getBaseDamage`; Ironclad-scoped ‚Äî keep both in sync with the card pool).
  - **OOC card choices** ‚Äî reward/shop/card-select append ` [Type, Rarity]` (e.g. `take card Offering [Skill, Uncommon]`) via `Card::getType()`/`getRarity()`.
- **Two parsers are coupled to these strings ‚Äî change them together.** (1) `rollout_view._PLAY_RE` tolerates the trailing `(deal ‚Ä¶)` so the target group doesn't swallow it. (2) `risk_proxies.classify_decision` strips a trailing `[‚Ä¶]` tag before matching the reward card name against `SELF_DAMAGE_CARDS`. If you change the `(deal ‚Ä¶)` or `[Type, Rarity]` format in the binding, update both. The enrichment **intentionally changes `state_text`/action-description content** (so LLM/heuristic trajectories shift); it does not affect the hybrid search agent. No `schemas.py` change (combat rides in free-form `state["combat"]`).
