# scripts

CLI entry points and build tooling. See the top-level [`CLAUDE.md`](../CLAUDE.md) for project goals and repo-wide rules.

## Contents

- `build_lightspeed.sh` — clones `sts_lightspeed`, applies `patches/sts_lightspeed_python_api.patch` (only if not already applied), and builds the `slaythespire` pybind module. Run this before any rollout.
- `run_rollout.py` — runs a single hybrid rollout from the CLI. Always invoke with `PYTHONPATH=src`.
- `run_batch.py` — batch rollouts over a seed range/list. Seeds come from `--seeds`, `--seed-start/--seed-count`, or `--seeds-config configs/frozen_seeds.json --split {smoke,dev,eval}`. For `--agent mlx` add `--thinking` (captures `<think>` chain-of-thought into `agent.thinking`), `--max-tokens`, `--temperature`. Use `--seed-timeout-seconds` for LLM/thinking runs (per-seed subprocess + kill) — thinking mode can hang on the seed-2-class UB and is slow.
- `run_sweep.py` — batched multi-model sweep: models × {thinking on/off} × seeds via the backend-appropriate cross-rollout orchestrator. `--backend mlx` uses `sts_ai.parallel_rollout` lockstep batching with `--batch-size` K; `--backend vllm` uses `sts_ai.streaming_rollout` continuous batching with `--concurrency` in-flight rollouts (effective concurrency is capped by the number of specs). Writes per-rollout JSONL + `.meta.json` under `<output-dir>/<agent_label>/`. One model load is reused across both thinking modes; a model that fails to load is skipped. `--enable-prefix-caching` is vLLM-only and defaults on. `run_rollout.py`/`run_batch.py` remain the serial path. **For `--backend vllm`, run ONE model per process** (vLLM does not free GPU memory between in-process loads): pass a single model per invocation and loop in the shell — `run_sweep` prints a warning and does only best-effort cleanup if given >1 model. **Measured per-model rollout speeds (for planning how long a run takes) are in [`docs/throughput_benchmarks.md`](../docs/throughput_benchmarks.md).**
- `run_until.py` — single-model full-game runner that launches exactly `--target` fresh rollouts, keeps N live (`--concurrency` for vLLM or `--batch-size` for MLX), and draws ascending seeds while skipping exclusions and existing outputs.
- `runpod/` — orchestration for running the sweep on rented NVIDIA GPUs (A40s): `setup_pod.sh` (apt deps + venv + `.[vllm]` + build sim), `run_sweep_on_pod.sh` (loops one model per `run_sweep.py` process, both reasoning arms, continue-on-failure; `CONCURRENCY` overrides the vLLM streaming in-flight cap, default 48), `sync_back.sh` (crash-resilient local rsync loop), per-pod model lists, and a `README.md` runbook (pod creation via the `runpod-spinup` skill, CUDA≥12.1 caveat, two-pod split, teardown). See [`runpod/README.md`](runpod/README.md).
- `compare_models.py` — per-model/mode comparison report over a sweep tree (outcomes, behaviour proxies incl. full-block-forgone & lethal-taken from `affordances`, reasoning length/latency from per-decision tokens, invalid-JSON rate).
- `summarize_rollouts.py` — JSONL rollouts → summary CSV.
- `compute_risk_proxies.py` — JSONL rollouts → risk-event CSV + aggregate JSON (deterministic, computed from stored traces).
- `build_sft_dataset.py` — offline filtered-BC dataset builder: a rollout label dir (`run_until`/`run_sweep` output) → `{prompt, completion}` SFT JSONL + a `.manifest.json`. Joins per-trajectory reward labels, filters to act-boss-clears (binary signal), and prints the positive count / stopped-reason breakdown / invalid rate. **Sparsity guardrail:** if positives fall below `--min-positives` it engages a top-floor-quantile fallback and **refuses to write** unless `--allow-fallback` is passed. Needs a tokenizer (`--tokenizer`, via `transformers`/`[llm]`/`[train-cuda]`). See [`src/sts_ai/CLAUDE.md`](../src/sts_ai/CLAUDE.md) "Training pipeline gotchas".
- `train_policy.py` — LoRA trainer dispatch: `--backend {mlx,trl}` over `src/sts_ai/train/train_mlx.py` (local mechanics, `mlx_lm lora` subprocess) or `train_trl.py` (CUDA, TRL `SFTTrainer`+PEFT). Consumes the `build_sft_dataset.py` JSONL+manifest; the resulting adapter is evaluated with `run_until.py --adapter-path`. Extras: `.[train-mlx]` / `.[train-cuda]`.
- `visualize_rollout.py` — Streamlit viewer for rollout traces (click-through / 1-decision-per-second autoplay). Out-of-combat decisions show deck/relics/potions + chosen action + reasoning + `<think>`; **combat decisions render a board** (enemy panels with HP/block/intent, a player panel with HP/block/energy/powers, and the hand as neutral card tiles with the chosen card + targeted enemy highlighted). Tiles are colour-by-state (chosen / unaffordable), **not by card type** — type isn't in the records (the "records only" data scope). Needs the `viz` extra. Run: `PYTHONPATH=src .venv/bin/streamlit run scripts/visualize_rollout.py`. All parsing lives in the unit-tested `sts_ai.rollout_view` (no Streamlit import there); the script is a thin render shell.

## Area gotchas

- **`--max-tokens` must be large for reasoning/thinking models (default 4096).** A small cap (e.g. 256) truncates the model mid-thought, so it never emits the closing JSON: `parse_json_action` fails, the decision is marked invalid, and after retries the rollout stops with `agent_invalid` before executing any fallback action. The eval's `invalid_rate` + `completion_tokens` (pinned at the cap) make this obvious. Raising the cap is **free for no-thinking models** (generation stops at EOS ~60-90 tokens, well under the cap), so the high default costs nothing there.
- **An unrecoverable invalid decision now *ends the rollout* (`stopped_reason=agent_invalid`), so outcome metrics are only interpretable beside the invalid rate.** Floor/win-rate from `summarize_rollouts.py` / `compare_models.py` partly reflect format-compliance, not just policy quality: an arm that truncates JSON more often will score lower for format reasons. Always read the `unexecuted` / `stopped_reason` columns (and `audit_rollouts.py`) next to outcome — never compare arms (esp. thinking vs no-thinking) on outcome alone. See `docs/rl_and_framing_design.md` gotcha 1.
- **`run_rollout.py` deletes the output file if it already exists** before writing. Pass a fresh `--output` path if you need to keep a prior trace.
- Records are appended per-decision during the run; a crash mid-rollout leaves a partial JSONL file.
- **Streamlit HTML rendering in `visualize_rollout.py` has two sharp edges.** (1) Markdown treats any line indented ≥4 spaces as a code block, so each HTML tile/panel must be emitted as a **single-line** string (no pretty-printed newlines) via `st.markdown(..., unsafe_allow_html=True)`. (2) A rerun (e.g. autoplay/`st.rerun`) re-renders the page from scratch, so the `<style>` block is **re-emitted every run** rather than once — don't guard it behind a session flag or the tiles lose their CSS.
